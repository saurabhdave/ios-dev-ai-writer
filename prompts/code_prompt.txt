## DEPRECATED API RULES
================================================================================

Do not present deprecated APIs as the recommended final pattern.
Deprecated or legacy APIs may appear ONLY inside an explicit `// ❌ Before`
block when the topic is about migration, deprecation, or modernisation.
Outside that block, use only the modern replacement.

Examples of deprecated APIs that must NEVER appear as the recommended final code:
  - `os_signpost` → use `OSSignposter` (available since iOS 15 / Swift 5.5)

## BEFORE/AFTER STRUCTURE RULES
================================================================================

Apply a before/after structure ONLY when the topic involves migration,
deprecation, or modernisation of an existing API or pattern:

  - Open with a `// ❌ Before` block showing the legacy/deprecated approach.
  - Follow with a blank line, then a `// ✅ After` comment block showing the
    modern replacement.
  - Each block must be self-contained and clearly labelled.

Example shape:
  // ❌ Before — ObservableObject (deprecated pattern)
  class OldViewModel: ObservableObject { @Published var count = 0 }

  // ✅ After — @Observable (Swift 6+)
  @Observable class ViewModel { var count = 0 }

For topics that are purely additive (a new API or new feature with no legacy
counterpart), omit the before/after structure entirely and show only the
modern pattern.

Example additive topic: "Swift Typed Throws"
→ Show only the modern typed-error pattern; no // ❌ Before block.

## SWIFT OBSERVATION RULES  (single source of truth)
================================================================================

When using Swift Observation (Swift 6+):
  - Use `@Observable` types instead of `ObservableObject`.
  - Use plain stored properties instead of `@Published`.
  - Use `@State` for view-owned observable instances.
  - Use `@Bindable` in the View — and ONLY in the View — when bindings ($)
    are required.
  - Always include `import Observation` when using `@Observable`.

CRITICAL — @Bindable placement (violating this causes validation failure):

  WRONG — never place @Bindable on properties inside @Observable:
    @Observable class MyModel { @Bindable var count: Int = 0 }

  CORRECT — plain stored property in the model; @Bindable only in the View:
    @Observable class MyModel { var count: Int = 0 }
    struct MyView: View { @Bindable var model: MyModel }

Never use `ObservableObject`, `@Published`, `@StateObject`, `@ObservedObject`,
or `@EnvironmentObject` outside a `// ❌ Before` block.
